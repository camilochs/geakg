# NS-SE Architecture

## Overview

NS-SE (Neuro-Symbolic Search Evolution) uses a 3-layer GEAKG (Generative Executable Algorithm Knowledge Graph) architecture:

```
┌─────────────────────────────────────────────────────────┐
│                    OFFLINE PHASE                        │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐             │
│  │   L0    │    │   L1    │    │   L2    │             │
│  │Topology │───▶│Operators│───▶│ ACO     │             │
│  │  (LLM)  │    │  (LLM)  │    │Training │             │
│  └─────────┘    └─────────┘    └─────────┘             │
│       │              │              │                   │
│       ▼              ▼              ▼                   │
│  [metagraph]   [pool.json]   [snapshot.json]           │
│  +weights      +operators    +pheromones               │
│  +conditions                 +symbolic_rules           │
└─────────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│                    ONLINE PHASE                         │
│  ┌──────────────────────────────────────────┐          │
│  │         Symbolic Executor                 │          │
│  │  (No LLM - pure symbolic reasoning)      │          │
│  │                                           │          │
│  │  - Operator selection by L2 pheromones   │          │
│  │  - Condition evaluation from L0          │          │
│  │  - Execute operators from L1             │          │
│  └──────────────────────────────────────────┘          │
└─────────────────────────────────────────────────────────┘
```

## GEAKG Concept

**Generative Executable Algorithm Knowledge Graph (GEAKG)** is a knowledge graph where:
- **Generative**: Structure and operators are generated by LLM
- **Executable**: Nodes contain executable code (not just descriptions)
- **Algorithm**: Represents algorithmic optimization strategies
- **Knowledge Graph**: Graph structure with roles, transitions, and learned weights

## Layers

### L0: MetaGraph Topology (Offline, LLM)

**Purpose**: Define the STRUCTURE of the knowledge graph
**Location**: `src/geakg/layers/l0/`

**Input**: Domain name + LLM
**Output**: `MetaGraph` with:
  - Nodes: 11 abstract roles (Construction, Local Search, Perturbation)
  - Edges: Transitions between roles
  - Initial weights: Heuristic preferences from LLM
  - Conditions: Adaptive control policies (e.g., "escape if stagnated > 10 iters")

**Key Files**:
| File | Description |
|------|-------------|
| `roles.py` | AbstractRole enum (11 roles in 3 categories) |
| `metagraph.py` | MetaGraph, MetaEdge classes |
| `conditions.py` | EdgeCondition, ExecutionContext |
| `patterns.py` | Predefined patterns (ILS, VNS) |
| `topology_generator.py` | LLM generates topology |

### L1: Operator Generation (Offline, LLM)

**Purpose**: Generate EXECUTABLE CODE for each role
**Location**: `src/geakg/layers/l1/`

**Input**: Roles from L0 + Base operators (A₀) + LLM
**Output**: `OperatorPool` with compiled, validated operators

**Process**:
1. Start with 11 base operators (one per role)
2. LLM generates variants using Design-Space Prompting
3. Mini F-Race selects statistically best performers
4. Pool saved for runtime use

**Key Files**:
| File | Description |
|------|-------------|
| `pool.py` | Operator, OperatorPool classes |
| `base_operators.py` | 11 base operators (A₀) |
| `generator.py` | L1Generator with AFO principle |
| `design_space.py` | Orthogonal design axes |
| `racing.py` | Statistical operator selection |

### L2: ACO Training (Offline, ACO)

**Purpose**: LEARN empirical knowledge through experience
**Location**: `src/geakg/layers/l2/`

**Input**: L0 (MetaGraph) + L1 (Pool) + Training instances
**Output**:
  - **Pheromones**: Refine/replace L0 heuristic weights based on successful paths
  - **Symbolic Rules**: Patterns extracted for online execution

**Key Files**:
| File | Description |
|------|-------------|
| `pheromones.py` | PheromoneMatrix, PheromoneEntry |
| `snapshot.py` | GEAKGSnapshot (L0 + L1 + L2 combined) |
| `symbolic_rules.py` | SymbolicRule, SymbolicRuleEngine |

---

## Phases

### Offline Phase: Training (L0 → L1 → L2)

**Location**: `src/geakg/offline/`

**Flow**:
1. **L0**: LLM generates MetaGraph topology with initial weights
2. **L1**: LLM generates operators for each role
3. **L2**: ACO trains on instances, learns pheromones, extracts rules

**Output**: `refined_pool.json` + `geakg_snapshot.json`

**Key File**: `iterative_refinement.py` - Multi-round training loop

### Online Phase: Execution (No LLM)

**Location**: `src/geakg/online/`

**Input**: GEAKGSnapshot (L0 + L1 + L2) + New instance
**Process**:
- Symbolic executor uses L2 pheromones for operator selection
- Evaluates L0 conditions for transitions
- Executes L1 operators on solutions

**Critical**: NO LLM calls at runtime - pure symbolic reasoning

---

## Transfer Learning

**Location**: `src/geakg/transfer/`

NS-SE supports cross-domain transfer where knowledge learned on TSP can be applied to other permutation problems.

### What Transfers

| Component | How It Transfers |
|-----------|-----------------|
| L0 Topology | Roles and transitions reused directly |
| L1 Operators | Code adapted via domain adapters |
| L2 Pheromones | Role preferences transfer |
| Symbolic Rules | Search strategy (stagnation thresholds, etc.) |

### Domain Adapters

Located in `src/geakg/transfer/adapters/`:
- `vrp_adapter.py` - Vehicle Routing Problem
- `jssp_adapter.py` - Job Shop Scheduling
- `pfsp_adapter.py` - Permutation Flow Shop
- `lop_adapter.py` - Linear Ordering Problem
- `qap_adapter.py` - Quadratic Assignment Problem

---

## Key Differences: L0 Weights vs L2 Pheromones

| Aspect | L0 (LLM) | L2 (ACO) |
|--------|----------|----------|
| **Origin** | Prior knowledge from LLM | Empirical experience |
| **Weights** | Initial, heuristic | Learned from paths |
| **Rules** | Predefined conditions | Discovered patterns |
| **Timing** | Before seeing instances | After training |
| **Updates** | Fixed once generated | Evolve with training |

## Usage Example

```python
from src.geakg.layers.l0 import MetaGraph, create_ils_meta_graph
from src.geakg.layers.l1 import OperatorPool
from src.geakg.layers.l2 import GEAKGSnapshot
from src.geakg.aco import MetaACOSelector, MetaACOConfig

# 1. Load trained GEAKG snapshot
snapshot = GEAKGSnapshot.load("experiments/knowledge/tsp_trained/geakg_snapshot.json")

# 2. Get components
metagraph = snapshot.get_metagraph()
pool = snapshot.get_operator_pool()
pheromones = snapshot.get_pheromone_matrix()

# 3. Configure ACO with learned knowledge
config = MetaACOConfig(
    enable_conditions=True,
    n_ants=15,
)

# 4. Run optimization
selector = MetaACOSelector(metagraph.instantiate("tsp"), config)
for iteration in range(100):
    ant = selector.construct_solution(problem_size=150)
    fitness = evaluate(ant.operator_path, instance)
    selector.update_pheromones_for_path(ant.role_path, fitness)
```

## Neuro-Symbolic Division

```
┌──────────────────────────────────────────────────────────────────┐
│                   THE NEURO-SYMBOLIC DIVISION                     │
│                                                                   │
│  NEURAL (LLM) - OFFLINE:                                         │
│    • Generates L0 topology (roles + transitions)                 │
│    • Generates L1 operators (executable code)                    │
│    • Assigns initial weights and conditions                      │
│    • One-time offline cost                                        │
│                                                                   │
│  SYMBOLIC (ACO/MMAS) - ONLINE:                                   │
│    • Navigates the MetaGraph using L2 pheromones                │
│    • Selects operators from L1 pool                              │
│    • Applies symbolic rules                                       │
│    • Makes ALL runtime decisions (no LLM calls)                  │
│                                                                   │
│  The LLM induces optimization GRAMMAR.                           │
│  The ACO discovers GOOD PATHS through that grammar.              │
└──────────────────────────────────────────────────────────────────┘
```
